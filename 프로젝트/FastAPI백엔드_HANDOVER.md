## 🚀 이번 단계: FastAPI 백엔드

### 왜 FastAPI인가
지금 챗봇은 **터미널에서만** 실행됨 (`python chatbot.py`). 웹/모바일에서 쓰려면 HTTP API로 노출해야 함. FastAPI가 그 역할.

### 계획서 요구사항 (사용자가 충족해야 하는 것)
- REST API 노출 (사용자 정보, 세션, 메시지)
- **스트리밍 응답** (계획서: 첫 토큰 5초 이내, 전체 15초 이내)
- 처리 과정 시각화 (어떤 도구·문서가 사용됐는지 프론트에 노출)
- 멀티 유저 지원 (`check_same_thread=False` 이미 적용됨)
- 다음 단계: React UI가 이걸 호출함

### 다음 작업 흐름 (제안 순서)
1. FastAPI 개념 짧게 (비유 → 개념) — 비전공자 친화적으로
2. 최소 동작 가능한 API 골격 (`POST /chat`, `GET /users` 정도)
3. 스트리밍 응답 (Server-Sent Events 또는 chunked) — 계획서 5초 토큰 요구
4. 처리 과정 노출 (tool_use 트레이스를 SSE 이벤트로)
5. 멀티 유저/세션 관리
6. CORS 설정 (React 호출 대비)
7. 배포 직전 환경변수·헬스체크

### 알려진 챗봇 이슈 (FastAPI 통합 시 의식할 것)
- 다중 호출 케이스에서 응답 33초까지 가는 경우 있음 → 스트리밍이 더 중요해짐
- 임베딩 모델 로딩 5~7초 → **서버 시작 시 한 번만 로딩** 해야 함 (요청마다 X)
- `FoodChatbot`, `FoodSearcher`, `GuidelineSearcher`, `HistoryStore` 인스턴스 재사용 필요

## 🎯 첫 응답에서 해야 할 것

이 인계 문서를 받았다면, **첫 응답에서**:

1. "프로젝트 상태 잘 확인했어. 정리하면 이런 상태야:" 로 시작
2. 현재 완성된 것 / 남은 것을 짧은 표로 정리
3. FastAPI를 비유 1개로 짧게 설명 (예: "지금 챗봇이 '터미널에서만 말하는 친구'라면, FastAPI는 그 친구한테 '전화기를 달아주는 작업'이야")
4. 작업 로드맵 (7단계 정도) 제시
5. 사용자에게 **다음 두 가지 중 어느 쪽인지** 물어보기:
   - 옵션 A: FastAPI 처음이니 개념부터 천천히
   - 옵션 B: 어느 정도 알아 — 바로 구현부터
6. 사용자가 **이미 가지고 있을 파일은 요청 안 함** (위 인벤토리에 있는 13개 코어 파일 모두 보유 중). FastAPI 시작 시 새 파일을 만들 거고, 기존 파일 중 `chatbot.py` 정도만 import해서 쓸 거임.

> **주의**: 코드를 한 번에 다 쏟아붓지 말 것. 한 단계씩 — 이게 사용자가 가장 좋아하는 방식.

## 📚 참고 — 이전 채팅방 진행 흐름

지난 채팅방에선 이런 흐름으로 진행했어 (참고용, 같은 톤 유지하면 좋음):

1. PDF 1개로 추출 → 청크 → 임베딩 → 검색 → 챗봇 통합 (End-to-end 1개 먼저)
2. 검증 후 13개 PDF 일괄 처리
3. 시스템 프롬프트 RAG 통합 튜닝
4. 평가 시나리오 10개 추가 → 25/25 통과 확인
5. HANDOVER.md / README.md 문서 정리

**각 단계마다**:
- 비유로 개념 설명
- 코드 하나씩 만들면서 핵심 포인트 4~5개 짚기
- 면접 단골 질문 짚기 (예: "왜 cosine distance 썼어요?")
- 실행 → 결과 확인 → 다음 단계

이 패턴 그대로 FastAPI에도 적용해주면 사용자가 익숙해서 잘 따라올 거야.
