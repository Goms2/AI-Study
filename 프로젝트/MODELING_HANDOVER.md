## 🔄 다음에 할 일 — 모델링 단계

### 모델링이란?

`food_name`(텍스트)을 **숫자 벡터로 변환**하는 작업. 챗봇이 "닭가슴살 저칼로리 추천"이라는 질문을 받으면, 의미가 비슷한 음식을 빠르게 찾을 수 있게 해주는 단계.

비유: 도서관 책에 **숫자 좌표**를 붙이는 작업. 좌표가 가까운 책 = 내용이 비슷한 책.

### 모델링 단계 작업 흐름

```
1. 임베딩 모델 선택 ← 가장 먼저 결정!
   ↓
2. 임베딩 대상 텍스트 설계 (food_name만? 영양정보도 같이?)
   ↓
3. 임베딩 파이프라인 구축 (배치 처리)
   ↓
4. 임베딩 결과 저장 (DB or 별도 파일)
   ↓
5. 품질 검증 (유사도 샘플 테스트)
```

### 임베딩 모델 선택지 (사용자에게 옵션으로 제시 권장)

| 옵션 | 장점 | 단점 | 비용 |
|---|---|---|---|
| **OpenAI text-embedding-3-small** | 성능 좋음, 다국어 지원 | API 비용 발생 | ~$0.02/1M tokens |
| **OpenAI text-embedding-3-large** | 최고 성능 | 비용 ↑, 차원 ↑ | ~$0.13/1M tokens |
| **한국어 특화 (KoSimCSE, KR-SBERT 등)** | 한국어 정확도, 무료 | 인프라 직접 운영 | GPU 필요 |
| **HuggingFace 다국어 모델** | 무료, 오픈소스 | 한국어 성능 다양 | GPU/CPU |

**제 추천 (보통)**: 음식 데이터가 한국어 위주이고 비용 부담 적다면 **OpenAI text-embedding-3-small** — 가성비 가장 좋음.

### 임베딩 텍스트 설계 (중요!)

`food_name`만 임베딩하면 "치킨"이 다 비슷하게 나옴. 다음과 같이 **컨텍스트를 합쳐서** 임베딩하면 정확도 ↑

```
예시: "후라이드 치킨, 한식, 구이·볶음, 1인분 250kcal, 단백질 25g"
```

옵션:
- **A) food_name만** — 빠름, 단순
- **B) food_name + 카테고리** — 균형
- **C) food_name + 카테고리 + 핵심 영양 (kcal/단백질)** — 추천 ⭐
- **D) 전체 영양 정보 다 포함** — 무거움

## 🎓 이전 채팅방에서 누적된 학습 패턴

### 매핑 작업 시 검증된 원칙들

1. **진단 먼저, 실행 나중** (SELECT → UPDATE)
2. **좁은 조건 먼저, 넓은 조건 나중**
3. **복합어 키워드 먼저, 단독 키워드 나중**
4. **`NOT LIKE`로 광범위 키워드 위험 차단**
5. **`=` 정확 매칭**: 단독 "굴", "대구" 같은 짧은 단어
6. **`%X류,%` 패턴**: 식약처 표준 형식 한 번에 잡음
7. **80/20 원칙**: NULL 5~10% 수용 (합리적 trade-off)
8. **카카오 산지명/외국어 = 초콜릿** 같은 도메인 지식 활용
9. **건강기능 분리**: 챗봇 활용도 핵심 (당뇨/임산부/어린이 추천)
10. **의도된 분류 우선순위**: 좁고 명확 → 광범위 순서
11. **백업 필수**: 매번 매핑 전 백업 테이블 생성
12. **식약처 분류 오류 인정**: NULL 유지가 정직한 경우 있음

### 사용자 작업 스타일

- 사용자는 결과 받으면 sub 이름을 **더 간결하게 수정**하는 감각이 좋음 (챗봇 친화 방향)
- JSON 파일로 결과 첨부하는 흐름 익숙함
- DB Browser에서 직접 실행 후 결과 텍스트로 공유
