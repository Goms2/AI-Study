## ❓ Q1. Transformer가 RNN보다 빠른 이유는 무엇인가요?

> 💡 **힌트**: "병렬 처리"와 "순차 계산"이 핵심 키워드. RNN은 t시점 계산에 t-1 시점 결과가 필요하다는 점을 생각해 보세요.

<details>
<summary>✅ 모범 답안 보기</summary>

<br>

**핵심 한 줄**: RNN은 **순차적**으로 계산해야 하지만, Transformer는 **모든 단어를 동시에(병렬로)** 계산할 수 있기 때문입니다.

### RNN의 구조적 한계

RNN은 시점 `t`의 hidden state를 계산하려면 시점 `t-1`의 결과가 반드시 필요합니다.

```
h_t = f(h_{t-1}, x_t)
```

즉, "I love you"를 처리하려면:

```
시간 →
[I] ────→ [love] ────→ [you]
   h_1 필요    h_2 필요
```

→ **GPU를 100개 줘도 한 줄로 줄 서서 처리해야 함** 😵

### Transformer의 병렬화

Transformer의 Self-Attention은 **모든 단어 쌍의 관계**를 행렬 곱셈 한 번으로 계산합니다.

```
Attention(Q, K, V) = softmax(QK^T / √d_k) V
```

이 연산은 **단어 순서에 의존하지 않으므로** GPU에서 완전히 병렬 처리 가능합니다.

### 시간 복잡도 비교 (논문 Table 1)

| 연산 종류 | 층당 복잡도 | 순차 연산 횟수 (병렬화 가능성) |
|-----------|------------|-------------------------------|
| **Self-Attention** | O(n² · d) | **O(1)** ← 병렬화 가능! |
| **Recurrent (RNN)** | O(n · d²) | O(n) ← 순차적 |
| **Convolutional** | O(k · n · d²) | O(1) |

> 📌 `n`은 시퀀스 길이, `d`는 차원
>
> Self-Attention은 시퀀스 길이가 짧을 때(`n < d`) RNN보다 층당 계산도 적고, **순차 연산이 O(1)** 이라 병렬화로 압도적으로 빠릅니다.

### 결과

논문에서는 8개 GPU로 **3.5일** 만에 SOTA를 달성했는데, 이는 기존 모델 대비 **훨씬 적은 학습 비용**입니다.

</details>

## ❓ Q2. Scaled Dot-Product Attention에서 왜 `√d_k`로 나누나요? 안 나누면 어떤 일이 생기나요?

> 💡 **힌트**: `d_k`가 커질수록 dot product 값의 분산이 어떻게 되는지, 그리고 그 값이 softmax에 들어갈 때 gradient에 어떤 영향을 주는지 생각해 보세요. (논문 Footnote 4 참고)

<details>
<summary>✅ 모범 답안 보기</summary>

<br>

**핵심 한 줄**: `d_k`가 커지면 dot product 값의 **분산이 커져서**, softmax 출력이 한 점에 극단적으로 쏠리고 **gradient가 거의 0**이 되기 때문입니다.

### 통계적 직관

Q와 K의 각 원소가 평균 0, 분산 1을 가진 독립 변수라고 가정하면, 둘의 dot product는:

```
q · k = Σ(q_i × k_i)   (i = 1 ~ d_k)
```

이때 통계적으로:
- **평균**: 0
- **분산**: `d_k` ← 차원에 비례해서 커짐!

→ `d_k = 64`면 표준편차가 약 `√64 = 8`. 즉, 값이 ±8 범위에서 들쭉날쭉.

### 큰 값이 softmax에 들어갈 때의 문제

softmax는 입력값 차이를 **지수함수적으로 증폭**합니다.

```python
# 예: d_k=512, 안 나누면
scores = [10, 11, 12]
softmax(scores) ≈ [0.09, 0.24, 0.67]  # 그래도 부드러움

# 하지만 d_k가 크면 값이 훨씬 크게 벌어짐
scores = [50, 60, 70]
softmax(scores) ≈ [0.0000, 0.0000, 1.0000]  # ← 한 점에 다 몰림!
```

이렇게 **한 점에만 몰리면(saturated)**, 다른 점들의 gradient가 거의 0이 되어 **학습이 안 됩니다.**

### √d_k로 나누면?

분산을 다시 1로 정규화하는 효과:

```
Var(q·k / √d_k) = d_k / d_k = 1
```

→ softmax 입력이 적당한 범위에 머물러 **gradient가 안정적으로 흐릅니다.**

### 논문 Footnote 4 (요약)

> "Q와 K가 평균 0, 분산 1인 독립 변수라면, 그 dot product `q·k`는 평균 0, 분산 `d_k`가 된다.
> 이를 보정하기 위해 `√d_k`로 스케일링한다."

> 📌 **한 줄 요약**: `√d_k` 스케일링은 **softmax saturation을 방지**해 gradient를 살리는 안정화 장치입니다.

</details>

## ❓ Q3. Multi-Head Attention에서 head를 여러 개 쓰는 이유는? 그냥 d_model 차원으로 한 번만 하면 안 되나요?

> 💡 **힌트**: "representation subspace"라는 개념과, 평균을 내면 잃어버리는 정보가 무엇인지 생각해 보세요. 각 head가 문법/의미 중 다른 것을 학습한다는 논문의 관찰도 연관 있습니다.

<details>
<summary>✅ 모범 답안 보기</summary>

<br>

**핵심 한 줄**: 한 번의 Attention은 **하나의 평균(mixture)** 만 만들어내지만, 여러 head를 쓰면 **서로 다른 관점(subspace)** 에서 동시에 관계를 학습할 수 있기 때문입니다.

### 한 번만 하면 생기는 문제: "정보 평균화"

Single-Head Attention은 본질적으로 **가중 평균**입니다.

```
output = Σ (attention_weight × Value)
```

문제는, 한 단어가 다른 여러 단어와 **다른 종류의 관계**를 가질 수 있다는 점입니다.

```
"The animal didn't cross the street because it was too tired."

"it"이 가리키는 것:
- 문법적 관점: 주어와 동사 일치
- 의미적 관점: "animal"을 가리킴 (지시 대명사)
- 거리적 관점: "tired"와의 인접성
```

한 번만 attention을 하면 → 이 모든 관점이 **하나의 가중 평균**으로 뭉개져서 미묘한 차이를 잃습니다.

### Multi-Head의 해결책: "Representation Subspace"

각 head는 서로 다른 **선형 변환(W_q, W_k, W_v)** 을 가집니다.

```
Head 1: 문법 관점에 특화된 Q, K, V로 attention
Head 2: 의미 관점에 특화된 Q, K, V로 attention
Head 3: 위치 관점에 특화된 Q, K, V로 attention
...
Head 8: 또 다른 관점
```

결과를 concat하면 **8가지 다른 subspace의 정보가 모두 보존**됩니다.

### 왜 d_k를 줄이나? (`512 → 64 × 8`)

각 head의 차원을 `d_k = d_model / h = 64`로 줄여도, **전체 계산량은 single-head와 비슷**합니다.

| 방식 | 차원 | 계산량 | 표현력 |
|------|------|--------|--------|
| Single-Head (d=512) | 512 | 동일 | 1가지 관점 |
| Multi-Head (d=64 × 8) | 64 × 8 | 동일 | **8가지 관점** ✨ |

→ **공짜로 표현력을 8배** 늘린 셈!

### 논문의 실증

논문 Figure 3-5에서 시각화한 attention 패턴을 보면, 각 head가 실제로 다른 언어학적 패턴(공동참조, 구문 구조 등)에 집중하는 것을 확인할 수 있습니다.

> 📌 **한 줄 요약**: Multi-Head는 **"여러 전문가가 각자 다른 관점에서 문장을 분석한 후 종합"** 하는 구조입니다.

</details>

## ❓ Q4. Positional Encoding을 왜 sin/cos로 만들었나요? 그냥 학습 가능한 임베딩(learned embedding)으로 해도 되지 않나요?

> 💡 **힌트**: 논문에서는 두 방식이 "거의 동일한 성능"이라고 했지만 sin/cos을 선택한 이유가 있습니다. 학습 시 본 적 없는 긴 문장에 대해 어떤 방식이 유리할지, 그리고 "상대적 위치"를 표현하는 데 sin/cos의 수학적 특성(선형 변환으로 shift 표현 가능)이 어떤 장점이 있는지 생각해 보세요.

<details>
<summary>✅ 모범 답안 보기</summary>

<br>

**핵심 한 줄**: sin/cos 방식은 ① **학습 시 보지 못한 긴 시퀀스로 외삽(extrapolation)** 이 가능하고, ② **상대적 위치를 선형 변환으로 표현**할 수 있다는 두 가지 수학적 장점이 있습니다.

### 두 방식 비교

| 방식 | 작동 원리 | 장점 | 단점 |
|------|----------|------|------|
| **Learned Embedding** | 위치별로 학습되는 벡터 (`nn.Embedding(max_len, d_model)`) | 데이터에 최적화됨 | 학습 시 본 길이까지만 처리 가능 |
| **Sinusoidal (sin/cos)** | 수학 공식으로 결정 | **임의 길이로 일반화 가능** | 학습 안 됨 (고정값) |

### 장점 1: 길이 외삽(Extrapolation) 가능

Learned Embedding은 학습 시 `max_len = 512`로 정했다면, 추론 시 600 길이 문장은 처리 못 합니다 (학습된 위치 벡터가 없으므로).

반면 sin/cos은 **공식이라 어떤 길이든 즉시 계산** 가능:

```python
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

`pos = 9999`든 `pos = 10000`이든 그냥 공식에 넣으면 끝.

### 장점 2: 상대적 위치를 선형 변환으로 표현

sin/cos의 핵심 수학적 성질:

```
PE(pos + k) = M_k · PE(pos)
```

즉, 위치 `pos+k`의 인코딩은 위치 `pos`의 인코딩에 **고정된 행렬 M_k**를 곱한 것으로 표현 가능합니다. (삼각함수의 덧셈 정리에서 유도)

**왜 중요한가?**
- 모델이 "k만큼 떨어진 단어"라는 상대적 관계를 **학습하기 쉬워집니다.**
- "5칸 떨어진 단어"를 인식하는 게 절대 위치를 외우는 것보다 일반화에 유리.

### 다양한 주파수의 의미

`10000^(2i/d_model)`로 차원마다 주파수를 다르게 줍니다:

```
낮은 차원 (작은 i): 빠르게 진동 → 인접한 위치 구분
높은 차원 (큰 i):  느리게 진동 → 멀리 떨어진 위치 구분
```

→ 마치 **시계의 초침/분침/시침**처럼 다양한 시간 스케일을 동시에 표현!

### 논문의 실증

논문 Table 3 (행 E)에서 두 방식을 비교한 결과:
- 거의 동일한 BLEU 점수
- → **학습 비용 + 일반화 능력**을 고려해 sin/cos 선택

> 📌 **한 줄 요약**: 성능은 비슷하지만, sin/cos은 **"공짜로 얻는 길이 일반화"** 보너스가 있어서 채택됐습니다.

</details>

## ❓ Q5. Decoder의 Self-Attention에는 Mask가 필요한데, Encoder에는 없습니다. 왜 그럴까요?

> 💡 **힌트**: Encoder는 입력 전체를 이미 다 알고 있지만, Decoder는 자기회귀적(auto-regressive)으로 한 단어씩 생성합니다. 훈련 시와 추론 시의 차이를 생각해 보세요. "커닝 방지"라는 비유도 도움이 됩니다.

<details>
<summary>✅ 모범 답안 보기</summary>

<br>

**핵심 한 줄**: Encoder는 **입력 전체가 이미 주어진 상태**이지만, Decoder는 **한 단어씩 생성**해야 하므로 훈련 시 미래 단어를 미리 보면 **"커닝"** 이 되어 추론 시 동작이 깨지기 때문입니다.

### Encoder vs Decoder의 역할 차이

| 구분 | Encoder | Decoder |
|------|---------|---------|
| **입력** | 원문 전체 (예: "I love you") | 지금까지 생성한 출력 (예: "나는 너를") |
| **목적** | 문장 이해 (양방향 컨텍스트) | 다음 단어 예측 (단방향, 순차적) |
| **모든 단어를 봐도 되나?** | ✅ 예 (입력은 정해져 있음) | ❌ 아니오 (미래 단어는 아직 없음) |

### 추론(Inference) 시의 Decoder 동작

Decoder는 **자기회귀(auto-regressive)** 방식으로 한 단어씩 생성합니다.

```
Step 1: [<start>]            → "나는" 예측
Step 2: [<start>, "나는"]     → "너를" 예측
Step 3: [<start>, "나는", "너를"] → "사랑해" 예측
...
```

각 단계에서 **이전에 생성한 단어들만** 보고 다음 단어를 예측합니다. 미래 단어는 **존재하지 않으므로** 볼 수 없습니다.

### 훈련(Training) 시의 문제

훈련 시에는 효율성을 위해 **정답 시퀀스 전체를 한 번에** Decoder에 넣습니다 (Teacher Forcing).

```
입력 (한 번에): [<start>, "나는", "너를", "사랑해"]
정답:          ["나는", "너를", "사랑해", <end>]
```

이때 Mask가 없으면 어떤 일이?

```
"나는"을 예측해야 하는 시점에서
Self-Attention이 "너를", "사랑해"까지 다 봐버림 → 정답 컨닝!
```

→ 모델이 **"앞 단어들만 보고 다음 단어 예측하는 능력"** 을 못 배우게 됩니다.
→ 추론 시에는 미래 단어가 없으니 **훈련-추론 불일치(train-test mismatch)** 로 성능 폭망 💥

### Mask의 작동 방식

미래 위치의 attention 점수를 `-∞`로 막아 → softmax 후 0이 되게 합니다.

```
Attention 점수 행렬 (4×4, 마스킹 후):

         나는    너를   사랑해   <end>
나는    [  0   -∞    -∞    -∞ ]   ← 자기 자신만 봄
너를    [  0    0    -∞    -∞ ]   ← 앞 단어까지만
사랑해  [  0    0     0    -∞ ]
<end>   [  0    0     0     0 ]   ← 다 볼 수 있음

         softmax 후:
나는    [ 1.0   0     0     0  ]
너를    [ 0.6  0.4    0     0  ]
사랑해  [ 0.3  0.4   0.3    0  ]
<end>   [ 0.2  0.3   0.3   0.2 ]
```

이걸 **Causal Mask** 또는 **Look-Ahead Mask**라고 부릅니다.

### Encoder는 왜 필요 없나?

Encoder는 입력 문장을 **이해**하는 역할이고, 입력은 이미 주어진 상태에서 시작합니다. 따라서:
- "love"가 "I"와 "you"를 모두 참고해도 OK (커닝이 아님)
- 양방향 컨텍스트를 활용하는 게 오히려 **문장 이해에 유리**

> 📌 **한 줄 요약**: Decoder Mask는 **"훈련 시에도 추론 시처럼 행동하게 만드는 장치"** 입니다 — 미래 단어 컨닝 방지!
