# BERT 면접 예상 질문 & 상세 답안

> 자연어처리 면접에서 BERT에 대해 자주 묻는 5가지 핵심 질문과 모범 답안

---

## Q1. BERT가 GPT와 가장 크게 다른 점은 무엇이며, 왜 그 차이가 중요한가요?

### 핵심 답변
BERT와 GPT의 가장 결정적인 차이는 **Self-Attention의 마스킹 방식**입니다.

| 구분 | GPT | BERT |
|---|---|---|
| 구조 | Transformer **Decoder** | Transformer **Encoder** |
| Attention | **단방향 (Causal Mask)** | **양방향 (No Mask)** |
| 사전학습 태스크 | 다음 토큰 예측 (Auto-Regressive LM) | MLM + NSP |
| 적합 태스크 | 생성 (Generation) | 이해 (Understanding) |

### 자세한 설명

**GPT는 Causal Mask를 적용**해서 `t` 시점의 토큰이 자신과 그 이전(왼쪽)의 토큰만 볼 수 있습니다. 다음 단어를 예측해야 하므로 미래 정보를 보면 정답이 누설되기 때문입니다.

**BERT는 마스킹 없이 모든 토큰이 모든 토큰을 참조**합니다. 한 단어의 의미는 양쪽 문맥 모두에 의해 결정되기 때문이죠. 예를 들어 "은행에 갔다"에서 "은행"이 금융기관인지 강가인지는 앞뒤 문맥을 모두 봐야 알 수 있습니다.

### 양방향이 결정적으로 유리한 태스크

1. **QA (질문 응답, SQuAD)**: 답의 시작/끝 위치를 찾을 때 답 토큰의 좌우 문맥이 모두 필요합니다.
2. **NER (개체명 인식)**: "워싱턴이 강을 건넜다"에서 "워싱턴"이 사람인지 도시인지는 뒤 문맥("강을 건넜다")까지 봐야 판단 가능합니다.
3. **NLI (자연어 추론)**: 전제-가설 관계를 판단하려면 두 문장의 모든 단어가 상호작용해야 합니다.
4. **문장 분류 (감정분석 등)**: 문장 전체의 의미를 종합해야 하므로 양방향 정보가 필수입니다.

### 한 줄 정리
> GPT는 "다음 단어 맞히기" 때문에 단방향일 수밖에 없고, BERT는 "빈칸 채우기"로 태스크를 바꿔서 양방향을 가능하게 만들었습니다. 이해 중심 태스크에서 양방향 문맥이 정확도를 크게 끌어올리는 것이 이 차이의 의의입니다.

## Q2. MLM에서 마스킹 비율이 15%이고, 그 안에서 다시 80/10/10으로 처리하는 이유는 무엇인가요?

### 핵심 답변
**사전학습(pre-training)과 파인튜닝(fine-tuning) 사이의 불일치(mismatch)를 완화하기 위함**입니다.

### 15% 마스킹 비율 선정 이유

- **너무 높으면 (예: 50%)**: 문맥 정보가 너무 적어져 빈칸을 추론할 단서가 부족해집니다.
- **너무 낮으면 (예: 5%)**: 한 번에 학습되는 토큰이 적어 학습 효율이 떨어지고, 수렴 속도가 느려집니다.
- **15%**: 실험적으로 충분한 문맥을 남기면서도 학습 효율이 좋은 균형점.

### 80/10/10 분할 처리

선택된 15% 토큰을 다시 다음과 같이 처리합니다.

| 비율 | 처리 방식 | 예시 (`my dog is hairy` → `hairy` 선택) |
|---|---|---|
| **80%** | `[MASK]` 토큰으로 교체 | `my dog is [MASK]` |
| **10%** | 랜덤한 다른 단어로 교체 | `my dog is apple` |
| **10%** | 원래 단어 그대로 유지 | `my dog is hairy` |

### 왜 이렇게 나누는가? — 핵심은 Mismatch 해결

#### ① `[MASK]`만 100% 사용한다면?
- 파인튜닝과 실제 추론 시점에는 **`[MASK]` 토큰이 등장하지 않습니다**.
- 모델은 사전학습 동안 `[MASK]` 위치만 신경 써서 예측하도록 학습되어, 일반 토큰에 대한 표현은 약해집니다.
- 결과적으로 사전학습과 다운스트림 태스크 사이의 입력 분포 차이로 인해 성능이 떨어집니다.

#### ② 10% 랜덤 토큰 교체가 필요한 이유
- 모델이 **"이 자리에 있는 토큰이 정말 옳은 토큰인가?"**를 항상 의심하도록 만듭니다.
- 즉, 입력으로 들어온 단어를 무조건 믿지 않고 문맥을 통해 검증하는 습관을 들입니다.
- 이는 **모든 토큰 위치에 대해 강건한 문맥적 표현을 유지**하도록 강제합니다.

#### ③ 10% 원래 단어 유지가 필요한 이유
- 모델이 "예측이 필요한 위치"만 골라내는 것이 아니라, **모든 위치의 표현을 학습**하도록 합니다.
- 또한 실제 추론 환경(마스킹 없는 일반 텍스트)과 가깝게 분포를 맞춥니다.

### 한 줄 정리
> `[MASK]`만 쓰면 모델이 `[MASK]` 위치에서만 작동하는 "전문가"가 되어 버립니다. 80/10/10 비율은 모델이 **어느 위치의 토큰이든 항상 문맥적 표현을 만들어내도록** 강제하는 장치이며, 이것이 사전학습-파인튜닝 mismatch를 줄이는 핵심 트릭입니다.

## Q3. BERT가 사용하는 3가지 Embedding은 무엇이며, 왜 합산해서 사용하는가요? Position Embedding은 Transformer 원논문의 sinusoidal과 어떻게 다른가요?

### 핵심 답변
BERT는 **Token + Segment + Position** 세 가지 임베딩을 element-wise 합산합니다.

### 3가지 임베딩의 역할

| 임베딩 | 담는 정보 | 차원 | 예시 |
|---|---|---|---|
| **Token Embedding** | 단어(WordPiece)의 의미 | (vocab_size, H) | `dog` → 의미 벡터 |
| **Segment Embedding** | 두 문장 중 어디에 속하는지 (A=0, B=1) | (2, H) | `[CLS] A1 A2 [SEP] B1 B2 [SEP]` |
| **Position Embedding** | 시퀀스 내 위치 (0~511) | (512, H) | 0번째, 1번째, ... 위치 |

### 왜 합산(sum)하는가?

세 임베딩은 모두 **같은 차원(H=768)을 가진 벡터**입니다.

1. **선형성과 효율성**: 단순 합산은 추가 파라미터 없이 세 정보를 결합하는 가장 간단한 방법입니다.
2. **고차원 공간의 분리 가능성**: 768차원이라는 충분히 큰 공간에서는, 서로 다른 종류의 정보가 직교에 가까운 부분공간(subspace)을 차지하도록 학습될 수 있습니다. 즉, 모델은 합쳐진 벡터에서 토큰 정보, 위치 정보, 세그먼트 정보를 효과적으로 분리해 사용할 수 있습니다.
3. **이후 LayerNorm 적용**: 합산 후 LayerNorm을 통해 분포를 정규화하므로 합산으로 인한 스케일 불균형 문제가 완화됩니다.
4. **Concatenate 대비 장점**: 이어붙이기(concat)를 하면 차원이 3배(2304)로 늘어나 파라미터가 폭증합니다. 합산은 그대로 H=768을 유지합니다.

### Sinusoidal vs Learned Position Embedding

| 구분 | 원조 Transformer (Sinusoidal) | BERT (Learned) |
|---|---|---|
| 생성 방식 | 사인/코사인 함수로 **고정 계산** | `nn.Embedding`으로 **학습** |
| 파라미터 | 없음 | `max_len × hidden_size` |
| 길이 외삽 | 임의 길이로 일반화 가능 | `max_len=512`로 **제한됨** |
| 장점 | 학습 데이터에 없는 긴 시퀀스도 처리 가능 | 데이터 분포에 최적화된 표현 학습 |
| 단점 | 데이터에 맞게 조정 불가 | 미리 정해둔 길이를 넘는 입력 처리 불가 |

#### Sinusoidal 공식 (참고)
```
PE(pos, 2i)   = sin(pos / 10000^(2i/d))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d))
```

### BERT가 Learned를 선택한 이유 (논문의 입장)
- 충분한 양의 데이터(BookCorpus + Wikipedia)로 학습하므로 위치 패턴을 직접 학습하는 것이 더 유연합니다.
- 사전학습 시에 어차피 512토큰으로 길이를 고정하므로 외삽 능력이 크게 필요하지 않습니다.
- 실제로 BERT 논문 실험에서 두 방식의 성능 차이는 크지 않았으나, 구현 단순성과 미세조정 가능성 측면에서 learned가 선택되었습니다.

### 한 줄 정리
> 세 임베딩은 각각 **의미 / 문장 구분 / 위치**라는 직교적 정보를 담고 있어 합산해도 분리 가능합니다. BERT는 sinusoidal 대신 learned position embedding을 써서 데이터 분포에 맞게 위치 표현을 최적화했지만, 그 대가로 512토큰이라는 길이 제한이 생겼습니다.

## Q4. `[CLS]` 토큰의 역할은 무엇이고, 사전학습이 끝난 직후의 `[CLS]` 벡터를 그대로 문장 임베딩으로 써도 되나요?

### 핵심 답변
`[CLS]`는 **분류 태스크용 문장 대표 토큰**이지만, **사전학습만 끝난 상태의 `[CLS]` 벡터는 의미 있는 문장 임베딩이 아닙니다**. 반드시 파인튜닝이 필요합니다.

### `[CLS]` 토큰의 역할

1. **시퀀스 맨 앞에 항상 추가되는 특수 토큰**: `[CLS] tok1 tok2 ... [SEP]`
2. **Self-Attention을 통해 모든 토큰의 정보를 종합**: 양방향 attention 덕분에 `[CLS]`는 시퀀스 전체와 정보를 주고받으며, 마지막 층에서는 문장 전체를 함축한 벡터가 됩니다.
3. **분류 헤드 입력**: 파인튜닝 시 `[CLS]`의 최종 hidden state(`C`)에 분류 레이어(`W_cls`)를 붙여 감정분석, NLI, NSP 같은 태스크를 수행합니다.

```
[CLS] → 마지막 층의 [CLS] 벡터 C → Linear(H, num_classes) → softmax
```

### 사전학습 직후의 `[CLS]`를 그대로 쓰면 안 되는 이유

BERT 원논문에서 명시적으로 언급된 사항입니다.

> "Note that without fine-tuning, the [CLS] vector C is not a meaningful sentence representation, since it is trained with NSP."

#### 이유 분석

1. **NSP는 단순한 이진 분류 태스크**:
   - "다음 문장인가 아닌가"라는 매우 거친 신호로만 학습됨.
   - 이 신호로는 의미적 유사도(semantic similarity), 토픽, 감정 등 풍부한 문장 표현이 학습되지 않습니다.

2. **`[CLS]`는 NSP에 특화된 표현**:
   - 사전학습 동안 `[CLS]`는 오직 NSP 헤드의 입력으로만 사용되므로, "두 문장이 연결되는지"를 판별하기 위한 특징만 골라서 담게 됩니다.
   - 일반적인 문장 의미 표현 측면에서는 정보가 부족합니다.

3. **실험적 증거**:
   - Sentence-BERT(Reimers & Gurevych, 2019) 논문에 따르면, 사전학습된 BERT의 `[CLS]`를 그대로 사용한 문장 유사도는 단순 GloVe 평균보다도 성능이 낮습니다.
   - Mean pooling이 `[CLS]`보다 종종 더 나은 결과를 보입니다.

### 해결책

| 상황 | 권장 방법 |
|---|---|
| 분류 태스크 (감정분석 등) | 파인튜닝 후 `[CLS]` 사용 ✅ |
| 문장 유사도 / 검색 (zero-shot) | **Sentence-BERT** 같이 contrastive 학습으로 추가 미세조정 |
| 일반 문장 임베딩 추출 | Mean/Max pooling이 `[CLS]`보다 나은 경우 많음 |

### 한 줄 정리
> `[CLS]`는 "분류 태스크의 입력으로 쓰일 자리"일 뿐, 사전학습 직후엔 NSP 편향이 강해 일반적 문장 임베딩으로는 부적합합니다. 의미 있는 문장 벡터가 필요하면 반드시 다운스트림 태스크로 파인튜닝하거나 Sentence-BERT 방식으로 추가 학습해야 합니다.

## Q5. BERT는 출력 토큰의 양방향 문맥을 사용하는데, 왜 일반적인 좌→우 언어모델로는 양방향 학습이 불가능한가요?

### 핵심 답변
**다층 구조에서 발생하는 정보 누설(self-seeing / information leakage) 문제** 때문에, 일반적인 LM 목적함수로는 깊은 양방향 학습이 불가능합니다. BERT는 **MLM(빈칸 채우기)** 으로 태스크 자체를 바꿔서 이 문제를 우회했습니다.

### 문제 상황: 다층 양방향 LM의 self-seeing

일반 LM의 목적은 `P(w_t | context)` 즉 **단어 예측**입니다. 만약 모든 층에서 양방향 attention을 허용하면 어떻게 될까요?

**1층(첫 Transformer 블록)**:
- `t` 시점의 hidden state `h_t^1`은 `w_t`를 비롯한 모든 입력 토큰을 본 결과입니다.
- 즉, `h_t^1`은 이미 자기 자신(`w_t`)의 정보를 포함합니다.

**2층**:
- `h_t^2`는 1층의 모든 `h^1`을 attention으로 참조합니다.
- 이때 `t`가 아닌 다른 위치 `s`의 `h_s^1` 안에도 **`w_t` 정보가 이미 들어있습니다**.
- 따라서 `h_t^2`는 직접 `w_t`를 보지 않더라도 **우회 경로(인접 위치)를 통해 자기 자신을 엿볼 수 있습니다**.

**결과**:
- 모델은 `w_t`를 예측하기 위해 문맥을 학습하는 대신, **양 옆 위치에 누설된 자기 정보를 그대로 베껴오는 trivial solution**을 학습합니다.
- 진짜 언어 이해 능력은 학습되지 않고, "정답 누설 경로 찾기"만 학습됩니다.

```
[w_1, w_2, w_3, w_4, w_5]
           ↑ w_3 예측
           
1층: h_3^1 = f(w_1, w_2, w_3, w_4, w_5)  ← w_3 정보 포함
2층: h_3^2 = g(h_1^1, h_2^1, h_4^1, h_5^1)  ← w_3 직접 안 봐도
                                              인접 h들에 w_3 정보가 누설됨
```

### 기존 시도들의 한계

#### ELMo (Peters et al., 2018)
- **단방향 forward LM**과 **단방향 backward LM**을 각각 따로 학습한 뒤, 두 representation을 concat 또는 가중합.
- 진정한 양방향이 아닌 **"shallow concatenation"** → 두 방향이 상호작용하지 못함.
- 각 LM은 단방향이므로 self-seeing 문제 없음, 하지만 깊은 양방향 통합은 불가.

#### 단순히 mask를 제거한 LM
- 위에서 설명한 self-seeing 문제 발생.
- 학습은 되지만 모델이 trivial solution을 학습해 성능 저하.

### BERT의 해결책: 태스크를 바꾸자 (MLM)

**핵심 아이디어**: 예측 대상 토큰을 **입력에서 제거**해버리면, 양방향으로 봐도 정답을 볼 수 없습니다.

```
원래: [w_1, w_2, w_3, w_4, w_5]에서 w_3 예측
      → 양방향 attention 하면 w_3가 누설됨

BERT: [w_1, w_2, [MASK], w_4, w_5]에서 [MASK] 자리 예측
      → 양방향 attention 해도 w_3 정보가 입력에 없음
      → 다른 단어들로부터 추론해야 함 ✅
```

이렇게 **태스크 정의 자체를 "다음 단어 예측"에서 "빈칸 채우기"로 바꿈**으로써:
- 자기 자신을 볼 수 없게 만들면서도
- 모든 층에서 자유롭게 양방향 attention 허용 가능
- 결과적으로 진정한 **deep bidirectional representation** 학습

### Q2와의 연결

여기서 다시 80/10/10이 등장합니다.
- `[MASK]` 토큰만 쓰면 파인튜닝과의 mismatch가 발생.
- 그래서 일부는 랜덤 토큰으로, 일부는 원래 단어로 두지만, **15%만 예측 대상**으로 정한 토큰들에 대해서만 loss를 계산하므로 self-seeing 문제는 여전히 해결됨.

### 한 줄 정리
> "다음 단어 예측" 목적함수는 다층 양방향에서 self-seeing이 발생하기 때문에 진정한 양방향 학습이 불가능합니다. BERT는 예측 대상을 **입력에서 가려버리는 MLM**을 도입해, 양방향 attention을 허용하면서도 정답 누설을 차단하는 우아한 해결을 제시했습니다.
